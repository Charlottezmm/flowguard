# FlowGuard v0.1 Agent Handoff

Last updated: 2026-05-24

This document summarizes the project context, decisions, implementation status,
and next steps from the v0.1 planning and build session. It is intended for the
next coding agent that works on this repository.

## Project Purpose

FlowGuard is a local workflow context bridge for AI coding agents.

It records workflow run facts, catches silent output failures, and produces
repair handoff artifacts that another coding agent can use without asking the
human to restate what happened.

Core positioning:

```text
Static project context: AGENTS.md / CLAUDE.md / Cursor rules / README
Dynamic run context: FlowGuard agent_context.md
```

`AGENTS.md` tells agents how to work in this repo.
FlowGuard's `agent_context.md` tells agents what happened in a workflow run and
what to fix next.

## Product Decisions

### Keep v0.1 Local And Deterministic

Do not add:

- hosted services
- MCP
- dashboards
- paid APIs
- external runtime dependencies
- LLM-generated diagnosis

FlowGuard should reduce uncertainty, not introduce another probabilistic layer.

### Runtime Facts First

The canonical v0.1 artifacts are:

```text
.flowguard/runs/latest/
  trace.json
  workflow_map.json
  agent_context.md
  outcome_report.html
```

Each explicit run should produce a clean `latest` directory.

### Repair Context Is The Core Artifact

`agent_context.md` is not a log summary. It is a dynamic repair handoff artifact
for coding agents.

It should answer:

1. Which workflow ran?
2. Which step failed?
3. What checks or errors failed?
4. What input and output did that step see?
5. Which upstream and downstream steps matter?
6. Which files should the coding agent read first?
7. What is the deterministic repair focus?

### Static Context vs Dynamic Context

The team discussed guidance around `CLAUDE.md`, `AGENTS.md`, Cursor rules, and
project memory. The useful takeaway is:

- static instructions define how an agent should work in a repo
- FlowGuard defines what happened in one run

Do not cite unverified attribution or statistics in README. Some external
claims discussed during planning were useful as product framing but were not
verified as primary sources.

## Implemented Behavior

### Runtime Core

Added:

- `flowguard_run(workflow_name)`
- clean `.flowguard/runs/latest` for explicit runs
- default latest artifact generation when steps run without explicit lifecycle
- step records with:
  - `id`
  - `name`
  - `status`
  - `duration_ms`
  - `source`
  - `input_summary`
  - `output_summary`
  - `failures`
  - `error`

Status values:

- `success`
- `failed`
- `error`

### Expectations

Available expectations:

- `valid_json`
- `required_fields`
- `min_count`
- `non_empty`
- `max_length`

### Workflow Map

`workflow_map.json` is generated from observed runtime order.

v0.1 deliberately does not do static analysis. Upstream is the previous observed
step and downstream is the next observed step.

### Agent Context

`agent_context.md` follows FlowGuard Repair Protocol v0.1:

- starts with `<!-- flowguard agent_context schema: v0.1 -->`
- reports the latest failed step only
- uses `Upstream:` and `Downstream impacted:`
- uses failed step source plus downstream source for relevant files
- caps input/output summaries at 1200 chars
- generates deterministic suggested focus text
- always exists, including no-failure runs

### Outcome Report

`outcome_report.html` is a local static HTML report:

- no web server
- no external CSS, JS, fonts, or assets
- HTML-escapes dynamic content
- shows step status, duration, upstream/downstream, output summary, failures,
  and a link to `agent_context.md`

## Current Demo Behavior

Run:

```bash
PYTHONPATH=src .venv/bin/python examples/github_issue_triage/pipeline.py
```

Expected generated files:

```text
.flowguard/runs/latest/trace.json
.flowguard/runs/latest/workflow_map.json
.flowguard/runs/latest/agent_context.md
.flowguard/runs/latest/outcome_report.html
```

The demo intentionally makes `issue.triage` fail because:

- `affected_files` is empty
- `repro_steps` has only one item

The resulting `agent_context.md` should direct a coding agent to update
`issue.triage` before `repair_brief.create` consumes its output.
