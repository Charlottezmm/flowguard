# FlowGuard v0.2 Initial Plan

Status: planning
Started: 2026-05-24

## Goal

FlowGuard v0.2 should make local workflow output checks more useful and make
run artifacts safer to compare over time.

The priority is not MCP first. The priority is to strengthen the local,
deterministic artifact layer that MCP or other integrations can later query.

## Recommended Direction

1. Add richer expectation/check results while preserving the current decorator
   API and v0.1 `agent_context.md` compatibility.
2. Add file artifact checks for workflows that produce local files.
3. Add API-like response checks for response objects already returned by a
   workflow step. FlowGuard should not make network calls.
4. Define golden run baselines for comparing important semantic fields across
   runs.
5. Add a local read-only query API for latest and golden artifacts.
6. Treat MCP as a thin read-only adapter after the local query API is stable.

## Candidate GitHub Issues

### `[v0.2] Define structured check result schema`

Current expectations return failure strings. v0.2 should introduce an internal
structured result shape that can support queries and comparisons without
removing the existing human-readable failure text.

### `[v0.2] Add file artifact checks`

Support checks for workflow outputs that reference local files:

- file exists
- file is non-empty
- file has allowed extension
- JSON file parses successfully

Path resolution rules must be explicit before implementation starts.

### `[v0.2] Add API-like response checks`

Validate response-like objects that workflow code already captured:

- status code range
- required JSON fields
- empty list detection
- error envelope detection

FlowGuard should not issue HTTP requests in v0.2.

### `[v0.2] Define golden run baseline format`

Define a deterministic baseline format for important run artifacts. Comparisons
should ignore unstable fields such as timestamps and durations.

### `[v0.2] Add golden run compare CLI`

Add local commands to create and compare golden baselines. The first version
should stay file-based and local-only.

### `[v0.2] Add local read-only query API`

Provide Python helpers for reading and summarizing artifacts, for example:

- load latest run
- find failed step
- summarize run status
- load agent context

This is the prerequisite for any MCP adapter.

### `[v0.2] Add minimal read-only MCP adapter`

If the local query API is stable, expose a minimal read-only MCP adapter for
latest run status, failed step, workflow map, and agent context. Do not add
workflow execution or write actions in the adapter.

### `[v0.2] Update FlowGuard skill for checks and golden runs`

Update the FlowGuard skill so agents know to run the workflow, inspect latest
artifacts, compare golden baselines when available, and then read
`agent_context.md`.

## Explicit Non-Goals

- No hosted service.
- No dashboard or web server.
- No LLM-generated diagnosis.
- No static call graph analysis.
- No workflow DAG editor.
- No external runtime dependency unless a concrete check cannot be implemented
  safely with the standard library.
- No breaking change to the v0.1 public API without a documented migration.

## Technical Risks

- `agent_context.md` has golden tests and should be treated as a protocol
  artifact. Text changes are protocol changes, not simple fixture churn.
- Failure strings are not a stable query interface. Structured check results
  should come before golden compare and MCP work.
- File checks need clear path semantics: current working directory, workflow
  source directory, and run directory are different anchors.
- Golden comparisons must normalize or ignore unstable fields such as
  `started_at`, `updated_at`, and `duration_ms`.
- MCP should not read `.flowguard/runs/latest` directly before a local query API
  exists; otherwise schema changes will leak into integration behavior.
