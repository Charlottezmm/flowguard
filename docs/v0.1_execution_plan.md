# FlowGuard v0.1 Execution Plan

This document is the source of truth for agents working on FlowGuard v0.1.
Every implementation agent should read this file before editing code.

FlowGuard is a local workflow context bridge for AI coding agents. v0.1 should prove that a Python workflow can record its run facts, catch silent output failures, and generate compact repair context that another coding agent can use immediately.

## Product Goal

FlowGuard v0.1 must make a local Python AI workflow explainable after a run.

The demo workflow should:

1. Run locally without API keys.
2. Produce deterministic local artifacts.
3. Catch an intentional silent failure in `issue.triage`.
4. Generate a concise `agent_context.md` that can be pasted into Codex, Claude Code, Cursor, or a similar coding agent.
5. Generate a simple `outcome_report.html` that a human can open locally.

## Current State

The repository already has:

- `src/flowguard/runtime.py` with a basic `@step` decorator.
- `src/flowguard/expectations.py` with basic expectations.
- `src/flowguard/artifacts.py` that writes `trace.json` and `agent_context.md`.
- `examples/github_issue_triage/pipeline.py` with a local demo workflow.
- `skills/flowguard/SKILL.md` with a first FlowGuard skill draft.
- GitHub issues `#1` through `#5` describing v0.1 work.

Known gaps:

- `.flowguard/runs/latest/trace.json` currently appends across runs, so the latest run can contain old steps.
- `workflow_map.json` is not implemented.
- `agent_context.md` is too thin compared with issue `#2` and README expectations.
- `outcome_report.html` is not implemented.
- Tests are not present.
- README and docs mention some artifacts that are not yet generated.

## Non-Negotiable Constraints

- Keep FlowGuard local-first.
- Do not add paid API calls.
- Do not add a hosted service.
- Do not add MCP in v0.1.
- Do not add a web server, React app, or dashboard.
- Do not add external runtime dependencies unless clearly justified.
- Keep the public API small and readable.
- Keep generated artifacts deterministic enough for tests.
- Preserve existing simple usage with `@step` and `@expect`.
- Prefer focused modules over a large framework rewrite.
- Use `PYTHONPATH=src python3` for local demo commands unless packaging is explicitly tested.

## Target Artifacts

Running the v0.1 demo must create:

```text
.flowguard/runs/latest/
  trace.json
  workflow_map.json
  agent_context.md
  outcome_report.html
```

The `latest` run must be clean for each demo execution. Running the demo multiple times must not append old steps into the current trace.

## Public API Target

The current API should remain valid:

```python
from flowguard import expect, step


@step("issue.triage")
@expect.required_fields(["issue_type", "severity", "repro_steps", "affected_files"])
@expect.min_count("repro_steps", 2)
def triage_issue(issue: dict) -> dict:
    ...
```

v0.1 should also support an explicit run lifecycle:

```python
from flowguard import flowguard_run


with flowguard_run("github_issue_triage"):
    issue = parse_issue(raw_text)
    triage = triage_issue(issue)
    create_repair_brief(triage)
```

If a step runs without an explicit `flowguard_run(...)`, FlowGuard should still work by creating or using a default `latest` run. The explicit run lifecycle is preferred for the demo and tests.

## Artifact Schema Targets

### trace.json

The trace should describe one workflow run:

```json
{
  "run_id": "latest",
  "workflow": "github_issue_triage",
  "started_at": "2026-05-12T13:00:00+00:00",
  "updated_at": "2026-05-12T13:00:01+00:00",
  "steps": [
    {
      "id": "issue.triage",
      "name": "issue.triage",
      "status": "failed",
      "duration_ms": 1,
      "source": "examples/github_issue_triage/pipeline.py",
      "input_summary": "{\"args\": [...], \"kwargs\": {}}",
      "output_summary": "{\"issue_type\": \"bug\", ...}",
      "failures": [
        "repro_steps must contain at least 2 items"
      ],
      "error": null
    }
  ]
}
```

`status` values should be:

- `success`
- `failed` for expectation failures
- `error` for exceptions

### workflow_map.json

The workflow map can be derived from the actual run order in v0.1. Do not implement broad static analysis.

```json
{
  "workflow": "github_issue_triage",
  "steps": [
    {
      "id": "issue.parse",
      "name": "issue.parse",
      "index": 0,
      "status": "success",
      "source": "examples/github_issue_triage/pipeline.py",
      "upstream": [],
      "downstream": ["issue.triage"]
    },
    {
      "id": "issue.triage",
      "name": "issue.triage",
      "index": 1,
      "status": "failed",
      "source": "examples/github_issue_triage/pipeline.py",
      "upstream": ["issue.parse"],
      "downstream": ["repair_brief.create"]
    }
  ]
}
```

### agent_context.md

The agent context must be short, deterministic, and directly useful for repair.

It should include:

- Workflow name.
- Failed step.
- Failed checks.
- Error, if present.
- Input summary.
- Output summary.
- Upstream steps when available.
- Downstream impacted steps when available.
- Relevant files when known.
- Suggested focus.

### outcome_report.html

The HTML report is for humans. It must be static and local.

It should include:

- Workflow name.
- Run status.
- Step table.
- Each step status.
- Duration.
- Failed checks.
- Output summary.
- Reference to `agent_context.md`.

Do not use external CSS, JavaScript, fonts, or remote assets.

## v0.1 Done Definition

v0.1 is complete when:

- A Python workflow can mark steps with `@step`.
- Outputs can be checked with `@expect`.
- Each explicit run produces a clean `latest` run directory.
- `trace.json` records the run facts.
- `workflow_map.json` records simple step order and dependencies.
- `agent_context.md` gives a coding agent enough context to repair the failed step.
- `outcome_report.html` gives a human a local readable summary.
- The GitHub issue triage demo runs without network access or API keys.
- The intentional `issue.triage` silent failure is reliably detected.
- Tests cover the runtime, expectations, workflow map, agent context, and report generation.
- README, `docs/design.md`, and `skills/flowguard/SKILL.md` match actual behavior.
