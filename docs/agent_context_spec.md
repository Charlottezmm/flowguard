# FlowGuard Agent Context Spec

`agent_context.md` is the FlowGuard Repair Protocol v0.1 artifact.

It is a dynamic repair handoff artifact, not a verbose log. It should be
concise, deterministic, and directly actionable for a coding agent.

## Relationship To Static Context

Static project context tells coding agents how to work in a repository:

- `AGENTS.md`
- `CLAUDE.md`
- Cursor rules
- project README files

FlowGuard dynamic run context tells coding agents what happened in one workflow
run and what to fix next:

- `.flowguard/runs/latest/agent_context.md`

## Schema Header

Every generated file starts with:

```md
<!-- flowguard agent_context schema: v0.1 -->
```

## Failure Selection

v0.1 reports the latest failed step only.

A failed step is any step in `trace.json` with either:

- non-empty `failures`
- non-empty `error`

If multiple steps fail, FlowGuard selects the last failed step in observed run
order and includes:

```text
Reporting: latest failed step
```

## Required Sections

For a failed run, `agent_context.md` includes these sections in order:

```text
Workflow
Run
Reporting
Failed step
Status
Failed checks
Error
Upstream
Downstream impacted
Relevant files
Input summary
Output summary
Suggested focus
Verification
```

`Error` is included only when the selected failed step has an exception.

## Terminology

Use:

```text
Upstream:
Downstream impacted:
```

Do not use `Upstream impacted`. Upstream steps provide context; downstream
steps are the steps impacted by a failed output.

## Relevant Files

Relevant files come from runtime facts only:

1. The selected failed step `source`.
2. Downstream step `source` values.
3. Duplicates removed.
4. Maximum 5 files.

v0.1 does not scan imports, inspect call graphs, or infer files through static
analysis.

## Summary Truncation

`Input summary` and `Output summary` are capped at 1200 characters in this
artifact even though `trace.json` already stores summaries.

When truncation occurs, append:

```text
...(truncated, full value in trace.json)
```

## Suggested Focus Templates

The suggested focus is deterministic. It must not be generated by an LLM.

Expectation failure with downstream:

```text
Update {failed_step} so its output satisfies the failed checks before {downstream[0]} consumes it.
```

Expectation failure without downstream:

```text
Update {failed_step} so its output satisfies the failed checks.
```

Exception:

```text
Inspect {failed_step} and its input summary. Fix the exception before changing downstream steps.
```

## No-Failure Runs

FlowGuard always generates `agent_context.md`, even when no step failed:

```md
<!-- flowguard agent_context schema: v0.1 -->

# Agent Context

Workflow: github_issue_triage
Run: latest

No failed checks in the latest run.
```

This keeps downstream tooling simple: agents can always read the same path.
