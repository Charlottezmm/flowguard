# FlowGuard v0.3.0 Release Review

Date: 2026-05-24
Release: `v0.3.0`
Release URL: <https://github.com/Charlottezmm/flowguard/releases/tag/v0.3.0>
Release commit: `25684cc`

## Release Result

v0.3.0 is shipped.

The release completed the planned v0.3 theme:

```text
integration experience + run comparison
```

The release was intentionally split into small PRs:

- PR #12: artifact schema versioning
- PR #13: skill intent workflow and docs
- PR #14: run comparison
- PR #15: golden and named-run path hardening
- PR #16: v0.3.0 release cleanup

## What Shipped

- Artifact-specific schema versions for stable JSON artifacts.
- Compatibility for reading v0.2 artifacts without `schema_version`.
- Fail-loud behavior for unknown future artifact schema versions.
- Stable-field documentation for `trace.json`, `workflow_map.json`, and golden
  `baseline.json`.
- FlowGuard skill guidance for check intent.
- Integration guide for adding FlowGuard to an existing Python workflow.
- Check cookbook for LLM output, API response, file artifact, JSON artifact, and
  downstream dependency failures.
- Named run save/list support under `.flowguard/runs/named/<workflow>/<name>/`.
- Run comparison for named runs, latest runs, and golden baselines.
- Agent-readable comparison diffs.
- Path hardening for named run and golden baseline references.
- Package and MCP metadata bumped to `0.3.0`.

## Release Boundary Check

v0.3.0 stayed within the intended product boundary:

- No hosted service.
- No dashboard.
- No workflow builder.
- No cloud sync.
- No write-capable MCP tools.
- No multi-language runtime.
- No restoration of `contracts.json` or `failed_contracts.md`.
- `trace.json` remains the source of truth for run facts.
- `agent_context.md` and `outcome_report.html` remain separate agent-facing and
  human-facing derived views.

## Issues Found

No release-blocking issues were found in the post-release review.

One issue was found before release finalization and fixed in PR #15:

- named run and golden baseline references needed path segment validation before
  filesystem writes or overwrites.

## Route Decision

Do not create a v0.4 by default.

The v0.3.0 release completed the intended bridge from `capture + judge` to
`capture + judge + compare`. The remaining work described in `docs/v1_scope.md`
is about freezing and validating the handoff loop, not adding another feature
release layer.

A v0.4 should only be introduced if a concrete post-release issue appears that:

- blocks v1.0 readiness,
- is too large or risky for a small hardening PR, and
- does not belong naturally inside a v1.0 stabilization PR.

Absent that evidence, the next planning target should be v1.0.

## Recommended v1.0 Discussion Frame

The v1.0 discussion should start from these questions:

1. What exactly must be frozen for the Repair Context Protocol?
2. What is the stable read-only MCP surface?
3. What clean-environment quickstart proves a stranger can use FlowGuard?
4. What real or near-real case study will demonstrate the full loop?
5. What README changes are needed before v1.0?
6. What visual proof, screenshot, GIF, or demo video is actually worth making?

The first v1.0 PR should likely be a planning/spec PR, not runtime code.
